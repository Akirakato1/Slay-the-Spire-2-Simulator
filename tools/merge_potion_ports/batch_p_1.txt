// Potion-effect data table — batch_p_1
//
// Survey of 64 potion .cs files in
//   sts2_decompiled/.../Core/Models/Potions/
// against the Effect VM vocabulary (effects.rs ~lines 47-555).
//
// Encoded = OnUse body whose primitive set (DealDamage / GainBlock / ApplyPower
// / DrawCards / GainEnergy / Heal / ApplyPower-multi / etc.) is already in the
// vocabulary AND uses no card-factory-from-pool / multi-step modal pick.
//
// Conventions used here:
// * TargetType.Self / TargetType.AnyPlayer    → Target::SelfPlayer
// * TargetType.AnyEnemy                       → Target::ChosenEnemy
// * TargetType.AllEnemies                     → Target::AllEnemies
// * `base.DynamicVars.Block` etc. canonical var keys:
//     BlockVar       → "Block"
//     DamageVar      → "Damage"
//     EnergyVar      → "Energy"
//     CardsVar       → "Cards"
//     RepeatVar      → "Repeat"
//     MaxHpVar       → "MaxHp"
//     StarsVar       → "Stars"
//     GoldVar        → "Gold"
//     ForgeVar       → "Forge"
//     PowerVar<XPow> → "XPower"  (the full type name; e.g. "VulnerablePower")
//     DynamicVar("Foo", N) / DynamicVars["Foo"] → "Foo"
//   This matches the convention already used in batch_v2_*.txt and batch_r_1.txt.
// * ValueProp.Unpowered flag on damage/block is intentionally dropped (matches
//   relic/card encoding cookbook): the Effect VM does not yet thread it through
//   the modifier pipeline. Several potions (BlockPotion, ShipInABottle, FirePotion,
//   ExplosiveAmpoule, PotionShapedRock, Fortifier) rely on this — flagged in
//   per-entry comments.
// * Unique custom power classes that are "the potion's own" — FlexPotionPower,
//   SpeedPotionPower, ShacklingPotionPower, BlockNextTurnPower, RetainHandPower —
//   are encoded with their literal C# type name. They're stub-applied via
//   ApplyPower; data layer only authors the call, runtime decides whether the
//   power-VM has a body for them.
